# Desktop Extension (DXT) Package — Design Spec

**Date:** 2026-06-11
**Status:** Approved (brainstorming phase)
**Owner:** juanpablo@vinixcode.com
**Relates to:** [Wasapi MCP Server v0.1](./2026-06-10-wasapi-mcp-design.md), [Setup Wizard v0.2](./2026-06-11-setup-wizard-design.md)

## 1. Purpose

Let users install the Wasapi MCP server in Claude Desktop **without ever opening a terminal**. Anthropic's Desktop Extension (DXT) format is a `.dxt` file (zipped MCP bundle) that the user double-clicks. Claude Desktop shows a native install dialog with a form for the API key, sets up the server, and starts running.

This complements the CLI wizard (which targets developers comfortable with terminals) and broadens the addressable audience to non-technical Wasapi users.

## 2. Non-goals (v0.3.0)

- Auto-discovery in any MCP marketplace (those marketplaces are still nascent; we can list later).
- Support for clients other than Claude Desktop (Cursor and others don't consume DXT files).
- Automated CI release via GitHub Actions. Manual `npm run package:dxt && gh release create` for v0.3.0; automate in a later iteration.
- Multi-language manifest (English only).
- DXT auto-update mechanism (the user re-downloads when they want a new version).
- Bundled Node runtime override (Claude Desktop's embedded Node handles execution).

## 3. Approach: self-contained bundle

The `.dxt` ships with everything the MCP server needs to run:
- The compiled `dist/` output (TypeScript → JavaScript)
- `node_modules/` with only runtime dependencies (`--omit=dev`)
- `manifest.json` (DXT-required metadata + user-input declarations)
- `icon.png` (300×300, Wasapi brand)
- `README.md` (shown in the install dialog)

**Why self-contained:**
- Works on machines that don't have Node/npm globally installed (Claude Desktop's embedded Node runs the bundle).
- No first-run network delay (npm download).
- Version is frozen at install time — predictable behavior, no surprise upgrades.
- Anthropic recommends bundled packages for production DXT.

**Trade-off accepted:** larger file (~3-7 MB) vs ~5 KB for a thin manifest that fetches from npm. Acceptable for a once-per-install download.

## 4. `.dxt` archive layout

```
wasapi-mcp-<version>.dxt   (ZIP)
├── manifest.json
├── dist/                      # tsc output (~30 KB)
│   ├── index.js               # entry point with shebang stripped (DXT runs via embedded node)
│   ├── server.js
│   ├── tools/
│   ├── lib/
│   └── setup/                 # included even though unused at runtime — small, harmless
├── node_modules/              # pruned: only runtime deps
│   ├── @modelcontextprotocol/sdk/
│   ├── @wasapi/js-sdk/
│   ├── zod/
│   ├── zod-to-json-schema/
│   └── <transitive deps>
├── icon.png                   # 300×300 PNG, Wasapi brand
└── README.md
```

Expected size: 3-7 MB packed. Sanity ceiling: 15 MB (smoke checklist flags anything larger).

### User input strategy (decision recap)

Only **one form field** in the install dialog: `WASAPI_API_KEY`. No `WASAPI_FROM_ID`. Rationale:
- Non-technical users don't know what a `from_id` is.
- Most accounts have one or two WhatsApp numbers — Claude can discover via `list_whatsapp_numbers` and use it implicitly.
- For accounts with multiple numbers, Claude asks the user in chat: *"¿desde qué número?"* — native conversational UX, no config editing.

The `sensitive: true` flag tells Claude Desktop to mask the field in the form and store the value in the OS keychain (not plaintext config).

### Field generation rules (`generate-manifest.mjs`)

- `version` ← `package.json#version`
- `author` ← `package.json#author` (object form)
- `homepage`, `documentation`, `support` ← derived from `package.json#repository` URL
- Other fields hard-coded in the script (description, long_description, keywords, server, user_config, tools, compatibility)

The script is pure (input: package.json contents → output: manifest object). Testable.

## 6. Build pipeline

### `scripts/build-dxt.mjs`

Node script (Node-pure for portability over bash). Steps:

1. `rm -rf release/`
2. `rm -rf dist/ && npm run build`
3. Delete `dist/**/.DS_Store`
4. Create `release/wasapi-mcp/`
5. Copy `dist/` → `release/wasapi-mcp/dist/`
6. Copy `package.json` → `release/wasapi-mcp/package.json` (temporary, for `npm install`)
7. Run `npm install --omit=dev --no-package-lock --silent` inside `release/wasapi-mcp/` to produce a pruned `node_modules/`
8. Delete `release/wasapi-mcp/package.json` (no longer needed)
9. Copy `assets/icon.png` → `release/wasapi-mcp/icon.png`
10. Copy `README.md` → `release/wasapi-mcp/README.md`
11. Invoke `generate-manifest.mjs` to write `release/wasapi-mcp/manifest.json`
12. Create ZIP: `release/wasapi-mcp-<version>.dxt` from contents of `release/wasapi-mcp/`
13. Print: archive path + size + file count

### npm script

```json
"scripts": {
  "package:dxt": "node scripts/build-dxt.mjs"
}
```

### `.gitignore` addition

```
release/
```

### Portability

`npm install` works everywhere. ZIP creation: use Node's built-in `node:zlib` + `node:fs` via a thin lib (avoid shelling out to `zip` for Windows compatibility). Dependency-free Node ZIP is doable in ~100 lines, or use `archiver` (popular, well-maintained). **Decision: use `archiver`** as a `devDependency` — adds ~50KB to dev install but saves writing/maintaining ZIP code.

## 7. Distribution

Manual flow per release:

```bash
npm version minor                            # 0.2.3 → 0.3.0
git push --follow-tags

npm publish --access public --otp=<code>     # npm

npm run package:dxt                          # builds release/wasapi-mcp-0.3.0.dxt

gh release create v0.3.0 \
  release/wasapi-mcp-0.3.0.dxt \
  --title "v0.3.0 — Desktop Extension support" \
  --notes-file release-notes.md              # or use --generate-notes
```

Always-latest install URL for end users:
`https://github.com/jpabloe/wasapi-mcp-server/releases/latest`

### Future automation (out of scope for v0.3.0)

A GitHub Actions workflow could:
- Watch for new `v*` tags
- Run `npm ci && npm run build && npm run package:dxt`
- Create the release with the `.dxt` attached
- (Optionally) publish to npm

Move to this once v0.3.0 manual flow has been exercised a few times.

## 9. Testing

### Unit tests

| File | Coverage |
|---|---|
| `tests/unit/generate-manifest.test.ts` | The manifest builder is pure: input `package.json` object → output manifest object. Tests: version sync, author resolution, all 12 tools present, `user_config.api_key.sensitive === true`, server entry_point correct. |
| Embedded zod schema | Minimal schema for required DXT fields; ensures the generated manifest stays valid if someone edits the builder. |

### Smoke checklist (`docs/dxt-smoke.md`)

Pre-release manual checklist:

1. `npm run package:dxt` → produces `release/wasapi-mcp-X.Y.Z.dxt` without errors
2. `unzip -l release/wasapi-mcp-X.Y.Z.dxt` → contains `manifest.json`, `dist/`, `node_modules/`, `icon.png`, `README.md`
3. Size between 3-7 MB; flag if > 15 MB
4. macOS: double-click opens Claude Desktop with install dialog
5. Dialog shows Wasapi name, icon, description, 12 tools, single API-key field (masked)
6. Install with real API key → MCP appears in Claude Desktop's MCP list
7. In a chat: *"Lista mis números de WhatsApp"* → expected response
8. Uninstall from Claude Desktop → removed cleanly

### Out of scope

- Headless Claude Desktop install test (impossible)
- Re-testing the 12 tools (existing unit + integration tests cover them)
- Windows / Linux smoke (covered manually before those release channels open)

### Test count delta

~5-8 new unit tests. Repo grows from 116 → ~123.
